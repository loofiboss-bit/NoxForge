# NoxForge v3.0.0 Implementation Plan

**Repository:** `loofiboss-bit/NoxForge`
**Reviewed baseline:** `main` / `v2.0.0` at `259dfd8f47d1fbd301197b751a7d5d02ce1fe935`
**Target:** Fedora KDE 44, Plasma 6.7+, Qt 6.11, Wayland
**Plan status:** Ready for Codex implementation, one phase at a time

## Outcome

NoxForge v3.0.0 becomes a release that Fedora users can install, update, verify,
and remove through a normal package-managed path.

The visual identity remains **Industrial Precision**. v3 is a reliability,
distribution, and live-qualification release—not another visual rewrite.

The end state is:

- a clean checkout passes the same release gate on supported build environments;
- every pull request is checked automatically before merge;
- a Fedora RPM owns all installed files and never applies the theme automatically;
- the currently pending Plasma, KWin, cursor, sound, and SDDM flows are tested in
  a real Fedora KDE 44 Wayland environment;
- actual live defects receive narrow fixes and regression coverage;
- users have one documented installation, upgrade, verification, rollback, and
  uninstall path;
- GitHub and COPR artifacts are produced only from a tagged, fully qualified
  release candidate.

## Current State

### Completed and worth preserving

- Original Plasma 6 Look-and-Feel, Plasma Style, Aurorae, KWin switcher, color
  scheme, icon, cursor, sound, wallpaper, splash, logout, and SDDM packages.
- Native Qt 6 `QStylePlugin` with a shared token-driven visual language.
- Coverage for 43 Plasma widget families, 165 scalable icons, 170 optical icon
  variants, 96 physical cursors, 32 sounds, and three wallpaper resolutions.
- Strong structural tests, offscreen Qt/SDDM rendering, LTR/RTL scale coverage,
  reversible dry-run installers, and deterministic source archive logic.
- Conservative safety model: installation does not apply the theme, reset the
  panel, edit SDDM configuration, or restart Plasma.

### Partial or unverified

- The offscreen evidence is useful but does not close the live Plasma, KWin, or
  real SDDM checks in `docs/MANUAL_TESTING.md`.
- The user and system installers are tested in temporary roots, but the normal
  end-user path still requires a source checkout, a Qt build, and two separate
  install commands.
- The source archive is deterministic in its own implementation, but generated
  Aurorae `.svgz` output is not currently byte-stable across Python/zlib
  environments.

### Missing

- No `.github/workflows/` CI or release workflow exists.
- No RPM spec, package build test, COPR source configuration, or package-manager
  upgrade/removal contract exists.
- No real-session evidence manifest connects each manual check to the exact
  environment, result, and screenshot/log.
- No read-only diagnostic command exists for confirming component discovery and
  collecting a concise support report.

### Confirmed defects found during review

1. On Python 3.12.13, `python3 scripts/validate.py` fails because five committed
   Aurorae `.svgz` files differ only in the gzip header OS byte generated by
   `gzip.compress(..., mtime=0)`.
2. The test suite reports 42 passing tests and one error because repository
   validation calls the failing generator check.
3. Public metadata contains the stale URL
   `https://github.com/loofiboss/NoxForge` instead of
   `https://github.com/loofiboss-bit/NoxForge`.
4. `CMakeLists.txt` installs only the native Qt style plugin; it does not define
   a complete system staging/install contract for the data packages.
5. `scripts/install-system.sh` hardcodes `/usr/lib64`, which should not be the
   production package-location authority.

## Top Priority

**Restore a truthful, cross-environment reproducible release baseline.**

This outranks packaging and visual polish because v2.0.0 currently fails its own
documented validation gate on a clean supported environment. CI or RPM work
built before this is fixed would automate a known inconsistent baseline.

## Scope

### Included

- Cross-environment deterministic generation.
- Metadata integrity and a single authoritative release gate.
- GitHub Actions CI using a Fedora 44 build environment.
- A Fedora-native RPM and COPR-ready build path.
- Safe staged installation using CMake/RPM ownership.
- Live Fedora KDE 44 Wayland qualification.
- Targeted fixes for defects reproduced during live qualification.
- Read-only diagnostics and practical install/rollback documentation.
- Tagged GitHub release artifacts and an explicitly approved COPR publication.

### Explicitly excluded

- Another full visual redesign or palette change.
- Increasing icon, cursor, sound, or wallpaper counts without a reproduced gap.
- Plasma 5, Qt 5, X11-first, non-Fedora, Flatpak, AppImage, or Windows ports.
- Automatic theme application, panel reset, SDDM activation, cache deletion, or
  Plasma restart.
- A custom GUI installer or theme manager.
- KDE Store publication in v3. A Store-only data package would omit the native
  Qt plugin and system SDDM component and would create a confusing partial
  product unless a separate product boundary is designed later.
- Unrelated dependency upgrades or architecture rewrites.

## Phase 0 — Restore the Release Baseline

### Objective and user value

Make the repository's release claims true on a clean checkout before adding new
delivery machinery.

### Likely components

- `scripts/generate_visual_assets.py`
- `scripts/validate.py`
- `tests/test_phase3.py`
- `tests/test_phase9.py`
- public `metadata.json` and `metadata.desktop` files
- `README.md`
- `docs/COMPATIBILITY.md`
- `docs/IMPLEMENTATION_PLAN.md`

### Implementation tasks

1. Replace the environment-dependent gzip path with one canonical writer that
   fixes timestamp, filename, compression level, and gzip header fields.
2. Add a regression test that asserts:
   - generated bytes match committed `.svgz` files;
   - the gzip header is canonical;
   - decompressed SVG bytes match the corresponding `.svg`;
   - repeated generation is byte-identical.
3. Test the generator contract against the supported Python versions used by
   local development and CI.
4. Correct every stale repository URL and extend validation so metadata cannot
   regress to a wrong owner/repository.
5. Make v3's plan the canonical future scope while retaining the v2 documents as
   historical release evidence.
6. Re-run the complete local gate and record the exact results. Do not describe
   unavailable Qt/CMake checks as passed.

### Acceptance criteria

- `python3 scripts/validate.py` passes from a clean checkout.
- All Python tests pass with no skipped release-integrity checks.
- Running every generator with `--check` reports no drift.
- Re-running generators changes no tracked file.
- All public website metadata points to
  `https://github.com/loofiboss-bit/NoxForge`.
- The worktree remains clean after the phase gate.

### Verification

```bash
python3 scripts/validate.py
python3 -m unittest discover -s tests -v
python3 scripts/generate_visual_assets.py --check
git diff --exit-code
```

### Checkpoint

Commit only after all available checks pass.

Suggested commit:

```text
fix: restore reproducible release validation
```

## Phase 1 — Add Required Continuous Integration

### Objective and user value

Prevent a release-integrity, build, installation, or metadata regression from
reaching `main`.

### Likely components

- `.github/workflows/ci.yml`
- `scripts/release-check.py` or an equivalent single gate entrypoint
- `tests/`
- `docs/CONTRIBUTING.md`
- `README.md`

### Implementation tasks

1. Create a least-privilege GitHub Actions workflow for pull requests and pushes
   to `main`.
2. Run the authoritative gate inside a Fedora 44 container with explicit Bash
   as the shell.
3. Install only declared build/test dependencies and document them.
4. Run:
   - generator drift validation;
   - Python tests;
   - CMake configure and Ninja build;
   - CTest with failure output;
   - QML lint for supported QML surfaces;
   - user/system dry-run install checks;
   - two independent archive builds with checksum comparison.
5. Add a small Python-version determinism job or equivalent fixture coverage so
   the gzip regression cannot return unnoticed.
6. Upload failure evidence only when useful; do not commit new rendered evidence
   from every CI run.
7. Add a CI badge only after the workflow passes on `main`.
8. Configure the workflow as a required branch check after it proves stable.

### Acceptance criteria

- A clean pull request runs without write permissions or repository secrets.
- CI fails on generated drift, metadata drift, test failure, build failure,
  non-reproducible archives, or unsafe installer behavior.
- CI passes on the repaired Phase 0 baseline.
- Local and CI commands use the same release-gate entrypoint.
- No workflow publishes, tags, or uploads to COPR.

### Verification

```bash
python3 scripts/release-check.py
git diff --exit-code
```

Also verify one intentionally broken test on a temporary branch causes the
expected CI failure before reverting it.

### Checkpoint

Suggested commit:

```text
ci: enforce the Fedora 44 release gate
```

## Phase 2 — Create the Fedora Package Contract

### Objective and user value

Replace the source-build/two-installer end-user path with one package-managed
installation that DNF can upgrade and remove safely.

### Likely components

- `CMakeLists.txt`
- `packaging/noxforge.spec`
- `packaging/`
- `scripts/install*.sh`
- `scripts/uninstall*.sh`
- `tests/test_packaging.py`
- `docs/INSTALL_FEDORA.md`

### Implementation tasks

1. Define CMake install rules for every runtime component using standard GNU/Qt
   install directories and `DESTDIR` staging.
2. Keep the native Qt plugin architecture-specific. Let the build system resolve
   its actual Qt plugin directory instead of treating `/usr/lib64` as source
   truth.
3. Add an RPM spec for Fedora 44. Start with one coherent `noxforge` package; add
   a separate optional SDDM subpackage only if it materially improves safety or
   dependency handling.
4. Declare exact build and runtime dependencies.
5. Ensure the package:
   - owns only NoxForge paths;
   - never modifies a user's active KDE settings;
   - never applies the Global Theme;
   - never activates SDDM;
   - never resets a panel or restarts Plasma.
6. Keep source installers as developer/fallback tools, or clearly deprecate them
   for end users. Do not maintain two competing production install authorities.
7. Test RPM build, install, upgrade, verification, downgrade/rollback procedure,
   and removal in disposable Fedora environments.
8. Run `rpmlint` on both the source RPM and binary RPM and resolve actionable
   findings.

### Acceptance criteria

- `rpmbuild` produces an installable RPM from a clean tagged source tree.
- `rpm -qlp` contains only intended NoxForge files.
- Installing the RPM makes every component discoverable.
- Installation leaves active KDE, panel, cursor, icon, sound, and SDDM settings
  unchanged.
- Reinstall and upgrade are safe.
- Removing the RPM deletes only package-owned NoxForge files and preserves
  unrelated files and user configuration.
- The raw source archive remains reproducible.

### Verification

```bash
rpmbuild -ba packaging/noxforge.spec
rpmlint <source-rpm> <binary-rpm>
rpm -qlp <binary-rpm>
sudo dnf install ./<binary-rpm>
rpm -V noxforge
sudo dnf remove noxforge
```

Run install/remove verification only in a disposable VM, container, or explicit
test root—not on the maintainer's active desktop during automated work.

### Checkpoint

Suggested commit:

```text
feat: add package-managed Fedora installation
```

## Phase 3 — Qualify the Real Plasma Experience

### Objective and user value

Turn v2's structural confidence into evidence that the actual desktop works, and
fix only defects reproduced in the live environment.

### Likely components

- `docs/MANUAL_TESTING.md`
- `docs/evidence/v3/`
- `tests/test_live_evidence.py`
- component files implicated by reproduced defects

### Implementation tasks

1. Create a structured evidence manifest containing:
   - NoxForge commit and package version;
   - Fedora, Plasma, KWin, Frameworks, Qt, kernel, and session versions;
   - display scale, layout, panel edge, blur state, and RTL state;
   - result: `passed`, `failed`, `blocked`, or `not-applicable`;
   - linked screenshot/log and concise observation.
2. Replace the old test that requires every live check to remain `Pending` with
   validation of honest status and evidence.
3. Test first in a disposable Fedora KDE 44 Wayland user/VM with a documented
   rollback path.
4. Execute the existing live matrix:
   - Global Theme discovery and explicit application;
   - preservation of the existing panel;
   - optional compact layout on every panel edge;
   - Plasma popups/dialogs with blur effect enabled and disabled;
   - Qt controls at 100% and 140%;
   - keyboard-only navigation and visible focus;
   - RTL shell behavior;
   - runtime confirmation of no visible fallback artwork;
   - Aurorae active, inactive, maximized, hover, and pressed states;
   - Alt+Tab with long titles, multiple windows, and empty state;
   - icon clarity at required sizes;
   - cursors at 100%, 140%, and 200%;
   - sound event routing and volume restraint;
   - splash, logout, and lock-screen surfaces;
   - multi-monitor placement;
   - SDDM flows.
5. Test SDDM in test mode/VM before any real login-screen activation. Preserve a
   recovery path to a known-good SDDM theme.
6. For each failure:
   - capture evidence;
   - reproduce it twice;
   - implement the smallest component-level fix;
   - add structural/offscreen regression coverage where possible;
   - rerun the affected live case and the full automated gate.
7. Do not change the palette, geometry system, or asset inventory unless a live
   defect proves that change is required.

### Acceptance criteria

- Every available live case has a result and evidence.
- No critical login, desktop, panel, input, or rollback flow is failed or
  unexplained.
- All fixed live defects have regression coverage where technically possible.
- The automated gate still passes after visual corrections.
- Screenshot evidence uses the real component; illustrative mockups are labeled
  and cannot satisfy a live gate.

### Verification

```bash
python3 scripts/release-check.py
python3 -m unittest tests.test_live_evidence -v
git diff --exit-code
```

Manual results are verified against the evidence manifest and checklist.

### Checkpoint

Use one commit per coherent reproduced defect, then a final evidence commit.

Examples:

```text
fix(plasma): remove panel-edge seam in vertical layouts
fix(kwin): preserve focus marker in maximized Aurorae state
test: record Fedora 44 Wayland qualification evidence
```

## Phase 4 — Add Read-Only Diagnostics and User Documentation

### Objective and user value

Make installation problems diagnosable without asking users to manually inspect
KDE paths or change settings.

### Likely components

- `tools/noxforge-doctor`
- `tests/test_doctor.py`
- `docs/INSTALL_FEDORA.md`
- `docs/TROUBLESHOOTING.md`
- `README.md`

### Implementation tasks

1. Add a read-only `noxforge-doctor` command that reports:
   - installed NoxForge package/version;
   - discovery of Look-and-Feel, Plasma Style, color, Aurorae, icons, cursors,
     sounds, wallpaper, KWin switcher, Qt style plugin, and SDDM theme;
   - active component names when safely readable;
   - session, Plasma, Qt, scaling, and display summary;
   - missing or mixed-version components;
   - clear next action without changing the system.
2. Support human-readable output. Add machine-readable output only if it is used
   by tests or issue intake.
3. Redact or omit unnecessary personal paths, usernames, hostnames, and display
   contents.
4. Document:
   - DNF/COPR installation;
   - explicit theme selection;
   - upgrade;
   - verification;
   - rollback before removal;
   - uninstall;
   - cache troubleshooting as an explicit manual action, not an installer side
     effect;
   - how to attach diagnostic output to a bug report.
5. Make the RPM installation path primary. Keep source installation in a clearly
   labeled maintainer/developer section.

### Acceptance criteria

- The doctor command performs no write, apply, restart, or privilege operation.
- It returns success for a complete installation and a non-zero status with
  actionable messages for missing/mixed components.
- Tests cover complete, partial, absent, and staged-root installations.
- A new Fedora user can install, select, verify, roll back, and remove NoxForge
  using the documentation without building Qt manually.

### Verification

```bash
noxforge-doctor
python3 -m unittest tests.test_doctor -v

```

## Phase 5 — Release v3.0.0 and Prepare COPR Delivery

### Objective and user value

Publish only artifacts produced from the reviewed, tested, tagged source state
and provide a stable update channel for Fedora users.

### Likely components

- `.github/workflows/release.yml`
- `packaging/`
- `docs/releases/v3.0.0.md`
- `README.md`
- `VERSION` and generated metadata consumers

### Implementation tasks

1. Update the version from the single authoritative source and regenerate every
   metadata consumer.
2. Add a release workflow that:
   - runs the full required gate;
   - builds the deterministic source archive;
   - builds SRPM/RPM artifacts;
   - emits SHA-256 checksums;
   - attaches the exact evidence summary;
   - refuses mismatched tag, metadata, or artifact versions.
3. Keep publication behind an explicit manual approval/environment gate. A tag
   alone must not silently publish to external services during implementation.
4. Create COPR project configuration for the confirmed owner/namespace and
   Fedora 44 target.
5. Build a release candidate in COPR, install it in the live test environment,
   and rerun package and smoke checks.
6. Write release notes that distinguish:
   - reliability/distribution changes;
   - live-confirmed visual fixes;
   - known limitations;
   - migration from v2 source installation.
7. After explicit maintainer approval:
   - create the signed/annotated `v3.0.0` tag;
   - publish the GitHub release;
   - publish/enable the COPR release;
   - verify installation from the public repository.

### Acceptance criteria

- Tag, `VERSION`, CMake project version, metadata, archive, RPM, and release
  notes all say `3.0.0`.
- Source archive and RPM checksums match downloaded public artifacts.
- Public RPM installation, upgrade from v2, doctor check, rollback, and removal
  are verified on Fedora KDE 44.
- No release process applies or activates NoxForge.
- CI and live qualification are green before publication approval.

## Release Gate

v3.0.0 is complete only when all of the following are true:

- [ ] Clean checkout has no generator drift.
- [ ] Python validation and all unit tests pass.
- [ ] CMake/Ninja build and every CTest pass.
- [ ] Supported QML surfaces pass lint/runtime checks.
- [ ] Two independent source builds are byte-identical.
- [ ] CI passes from the exact release commit.
- [ ] SRPM/RPM build and `rpmlint` checks pass.
- [ ] Disposable RPM install, upgrade, verify, and removal checks pass.
- [ ] Package installation changes no active desktop or SDDM setting.
- [ ] Every available Fedora KDE 44 Wayland live case has evidence.
- [ ] No critical live case is failed, unexplained, or represented only by a
      mockup.
- [ ] Diagnostic output is read-only and actionable.
- [ ] Version and repository metadata are consistent.
- [ ] Release notes and rollback instructions are current.
- [ ] Publication receives explicit maintainer approval.

## Risks and Open Decisions

1. **COPR owner/namespace:** confirm the actual COPR account before writing public
   installation commands. Do not invent it.
2. **SDDM live testing:** use a VM/test mode and a documented recovery path before
   activation on a real login screen.
3. **RPM package split:** default to one coherent Fedora package. Split SDDM only
   if dependency/safety testing proves a clear benefit.
4. **Supported Python range:** define it from actual Fedora/CI environments and
   test the canonical gzip writer across that range.
5. **Live evidence storage:** keep evidence compact. Do not turn the repository
   into a large screenshot archive.
6. **KDE Store:** reconsider after v3 only if a clearly named data-only edition
   can coexist without users believing it includes the native Qt style and SDDM.
